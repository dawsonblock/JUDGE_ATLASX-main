# Proof Policy

This alpha release uses a self-verifying proof chain to demonstrate that:

1. **Code compiles** - backend_compile.log proves Python code is importable
2. **Tests collect** - backend_pytest_collect.log proves test infrastructure works
3. **Tests pass** - backend_pytest.xml proves 3,551 tests passing
4. **Frontend builds** - frontend_build.log proves Next.js build success
5. **Frontend tests pass** - frontend_test.log proves 104 tests passing
6. **Docker ready** - docker_runtime_preflight.log proves services can start
7. **Archive valid** - archive_validation.log proves structure is correct

Every proof log is generated by a real command during the release cycle.
No proof files are manually written or faked.

## Proof Files Are Not Mutable

Once generated, proof files cannot be changed without invalidating the release.
The archive SHA-256 checksum guarantees no modifications occurred after packaging.

## Validator Scripts

Run these to verify the proof chain is intact:

```bash
python3 scripts/check_required_proof_logs.py --strict-required-files
python3 scripts/check_proof_manifest.py
python3 scripts/check_proof_consistency.py
python3 scripts/check_no_local_paths_in_release_proof.py
```

All must return PASS.

## Alpha vs Production

This alpha release prioritizes transparency over production hardening.
Every claim is provable. Every proof is real. This is intentional.

For production, we will add enterprise security audits and legal review.
For now, this proof chain is the validation mechanism.
